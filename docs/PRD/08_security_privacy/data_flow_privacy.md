# Data Flow & Privacy Analysis

This document maps every data flow in the Voice & Vision Assistant for Blind, classifying what data leaves the local machine versus what stays local. It defines cloud exposure boundaries, data handling policies, and privacy controls.

---

## 1. Data That Leaves the Local Machine

The following data is transmitted to external cloud services during normal operation:

### 1.1 Audio Streams to Deepgram (STT)

- **Data Sent**: Raw audio stream from the user's microphone via WebSocket (WSS).
- **Protocol**: WebSocket Secure (WSS) with TLS encryption in transit.
- **Purpose**: Real-time speech-to-text transcription.
- **Data Received**: Text transcription of the audio.
- **Retention**: Governed by Deepgram's data processing agreement. No audio is stored locally by default.
- **Privacy Note**: Raw voice data contains biometric characteristics (voice print). Users should be aware that their voice is transmitted to Deepgram's cloud infrastructure.

### 1.2 Text Prompts to qwen3.5:cloud (LLM)

- **Data Sent**: Natural language prompts containing scene descriptions, user questions, memory context (for RAG queries), and system instructions. Sent via HTTPS to the Ollama cloud runtime.
- **Protocol**: HTTPS with TLS encryption in transit.
- **Purpose**: LLM reasoning for VQA answers, RAG-based memory queries, general chat responses, and memory summarization.
- **Data Received**: Natural language responses from the LLM.
- **Privacy Note**: Scene descriptions may reference people, objects, and locations observed by the camera. While raw images/frames are never sent, the textual descriptions can contain personally identifiable observations (e.g., "a woman in a red coat standing near the pharmacy counter").

### 1.3 Text to ElevenLabs (TTS)

- **Data Sent**: Plain text strings for speech synthesis via HTTPS/WebSocket.
- **Protocol**: HTTPS or WSS with TLS encryption in transit.
- **Purpose**: Converting text responses to natural-sounding audio for the blind user.
- **Data Received**: Audio stream (synthesized speech).
- **Privacy Note**: Text sent to ElevenLabs may contain answers referencing the user's environment, memories, or QR code content. No raw images or audio are sent.

### 1.4 WebRTC Media via LiveKit

- **Data Sent**: Bidirectional audio and video streams via WebRTC (SRTP-encrypted).
- **Protocol**: DTLS/SRTP for media encryption. WSS for signaling.
- **Purpose**: Real-time audio/video transport between the user's device and the assistant agent.
- **Data Received**: Audio output from the assistant.
- **Privacy Note**: Video frames from the user's camera traverse LiveKit's infrastructure for transport. While frames are processed locally by the agent, the WebRTC transport layer routes media through LiveKit's servers.

### 1.5 Optional Avatar Data to Tavus

- **Data Sent**: Audio stream and session configuration for virtual avatar rendering.
- **Protocol**: HTTPS with TLS encryption in transit.
- **Purpose**: Generating a visual avatar representation for video calls.
- **Data Received**: Avatar video stream.
- **Privacy Note**: Only enabled when `ENABLE_AVATAR=true`. Disabled by default. When active, audio data is shared with Tavus for lip-sync rendering.

### 1.6 Search Queries to DuckDuckGo

- **Data Sent**: Plain text search queries via HTTPS.
- **Protocol**: HTTPS with TLS encryption in transit.
- **Purpose**: Real-time information retrieval for user questions.
- **Data Received**: Search result snippets and URLs.
- **Privacy Note**: Search queries may reveal user interests and information needs. DuckDuckGo does not track users by design.

## 2. Data That Stays Local

The following data categories never leave the local machine:

### 2.1 Camera Frames

Raw camera frames (numpy arrays) are captured by the LiveKit agent and processed entirely on the local GPU. Frames are used for object detection (YOLO), depth estimation (MiDaS), OCR (EasyOCR), face detection, QR scanning (pyzbar), and braille recognition. Frames are held in the `LiveFrameManager` bounded deque and discarded after processing. No frame data is saved to disk by default (unless `save_raw=true` is explicitly set in a memory store request).

### 2.2 Perception Results

All outputs from the perception pipeline remain local:
- Object detections (bounding boxes, class names, confidence scores)
- Depth maps (per-pixel depth estimates from MiDaS or SimpleDepthEstimator)
- Segmentation masks (from EdgeAwareSegmenter)
- Scene graphs (structured spatial relationships between detected objects)
- Navigation cues (obstacle records with distance, direction, and priority)
- OCR text extracted from images

### 2.3 FAISS Embeddings and Vector Index

Text embeddings generated by `qwen3-embedding:4b` (384-dimensional vectors) are computed locally and stored in the FAISS IndexFlatL2 index at `data/memory_index/`. The embedding model runs entirely on the local GPU (~2GB VRAM). Neither the embeddings nor the index data are transmitted to any cloud service.

### 2.4 ONNX Model Weights

Pre-trained model weights stored in the `models/` directory (YOLO v8n, MiDaS v2.1 small) are loaded locally via ONNX Runtime. Model files are never uploaded or transmitted.

### 2.5 QR Cache

Decoded QR scan results are cached locally in `qr_cache/` as JSON files with configurable TTL (`QR_CACHE_TTL_SECONDS`, default 86400s). Cache entries contain decoded payloads, content classification, and timestamps. This data never leaves the machine.

### 2.6 Face Consent State

Face detection consent preferences are persisted to `data/face_consent.json`. This file contains the consent boolean, timestamp, and optional reason. It is never transmitted externally.

### 2.7 Session Logs

Structured telemetry and debug logs are stored locally in `.runtime/logs/`. Logs contain processing timestamps, performance metrics, and component status. No log data is forwarded to external services.

### 2.8 Depth Maps

Per-pixel depth maps generated by MiDaS v2.1 or SimpleDepthEstimator remain in-process. They are used by the SpatialFuser to compute obstacle distances and are discarded after the frame processing pipeline completes.

### 2.9 Scene Graphs

Structured scene graph data (object relationships, spatial positions) generated by SceneGraphBuilder stays local. Only the human-readable scene description (text summary) may be sent to qwen3.5:cloud as part of VQA or RAG prompts.

## 3. Cloud Exposure Boundaries

| Cloud Service | Data Sent | Data Received | Encryption in Transit | Frequency |
|---------------|-----------|---------------|----------------------|-----------|
| **Deepgram** (STT) | Raw audio stream via WebSocket | Text transcription | WSS (TLS) | Continuous during voice interaction |
| **qwen3.5:cloud** (LLM) | Text prompts (scene descriptions, user questions, memory context) | Text responses (answers, summaries) | HTTPS (TLS) | Per VQA query, per RAG query, per chat message |
| **ElevenLabs** (TTS) | Plain text for synthesis | Audio stream (synthesized speech) | HTTPS/WSS (TLS) | Per response utterance |
| **LiveKit** (WebRTC) | Bidirectional audio/video via WebRTC | Audio/video relay | DTLS/SRTP | Continuous during session |
| **Tavus** (Avatar) | Audio stream, session config | Avatar video stream | HTTPS (TLS) | Continuous when avatar enabled (optional) |
| **DuckDuckGo** (Search) | Text search queries | Search result snippets | HTTPS (TLS) | Per search intent |

## 4. Audio Data Handling

### 4.1 Inbound Audio (User Speech)

- **Source**: User's microphone captured via LiveKit WebRTC.
- **Processing**: Audio is streamed to Deepgram via WebSocket for real-time STT.
- **Local Storage**: No audio is stored locally by default. Audio can optionally be persisted in memory store requests when `save_raw=true` and `audio_base64` is provided, but this requires explicit consent.
- **Cloud Exposure**: Raw audio traverses LiveKit's WebRTC infrastructure (SRTP-encrypted) and is sent to Deepgram (WSS-encrypted) for transcription.

### 4.2 Outbound Audio (Assistant Speech)

- **Source**: Text responses generated by the assistant.
- **Processing**: Text is sent to ElevenLabs for synthesis. The `StreamingTTSCoordinator` buffers text at sentence boundaries before dispatch.
- **Local Storage**: No TTS audio is cached locally (caching recommended in BACKLOG-024 but not yet implemented).
- **Cloud Exposure**: Plain text is sent to ElevenLabs. Synthesized audio is returned and relayed to the user via LiveKit.

## 5. Image/Frame Handling

### 5.1 Privacy-by-Design Principle

Camera frames never leave the local machine in raw form. The system follows a strict privacy boundary:

1. **Camera capture**: Frames are captured by the LiveKit agent and stored in a bounded in-memory deque.
2. **Local processing**: All vision inference (detection, depth, segmentation, OCR, face detection, QR scanning) runs on the local GPU or CPU.
3. **Text extraction**: Only human-readable text summaries of the scene are generated.
4. **Cloud transmission**: Only the text description (not the image) is sent to qwen3.5:cloud for reasoning.

### 5.2 Frame Lifecycle

```
Camera → LiveFrameManager (bounded deque) → PerceptionWorkerPool (GPU inference)
    → SceneGraphBuilder (local) → Text description → qwen3.5:cloud (if VQA query)
```

- Frames are discarded from the deque when newer frames arrive (FIFO eviction).
- The 500ms freshness check ensures stale frames are not processed.
- No frame is written to disk unless explicitly requested via `/memory/store` with `save_raw=true`.

## 6. Embedding Privacy

### 6.1 Local Embedding Generation

The `qwen3-embedding:4b` model runs entirely on the local GPU (~2GB VRAM) via the Ollama local runtime. All embedding generation occurs on-device:

- **Input**: Text strings (memory summaries, search queries).
- **Output**: 384-dimensional float vectors.
- **Processing**: Synchronous HTTP call to the local Ollama service (localhost).
- **Network**: No external network calls. The Ollama embedding service runs as a local process.

### 6.2 Privacy Guarantee

Embeddings generated by `qwen3-embedding:4b` never leave the local machine. They are:
- Computed locally on the GPU.
- Stored in the local FAISS index (`data/memory_index/`).
- Queried locally using FAISS L2 distance search.

The only text that reaches the cloud is the natural language query and retrieved memory summaries sent to qwen3.5:cloud for RAG reasoning. The raw embedding vectors are never transmitted.

## 7. Memory Retention Policy

### 7.1 Consent-Gated Storage

- **Default State**: `MEMORY_ENABLED=false`. The memory system is opt-in only.
- **Consent Endpoint**: `/memory/consent` records explicit user consent with timestamp and optional reason.
- **Consent Persistence**: Consent state is persisted to file-backed storage, surviving process restarts.
- **Gate Enforcement**: All memory operations (`/memory/store`, `/memory/search`, `/memory/query`) check consent before proceeding. Requests without consent are rejected.

### 7.2 Configurable TTL

- Memory entries have an `expiry` timestamp set at creation time.
- Auto-expiry is supported: memories past their TTL are eligible for deletion.
- TTL values are configurable per deployment.

### 7.3 User-Initiated Deletion

- Users can export all data via `/export/full` and import via `/import/full`.
- The export includes all memories, QR cache, face consent state, and session data.
- Data deletion is achievable by clearing the FAISS index and associated metadata.

### 7.4 Raw Media Controls

- Raw image and audio data are not saved by default (`save_raw=false`).
- When `save_raw=true` is explicitly set in a `/memory/store` request, raw media is persisted alongside the memory entry.
- Users control raw media retention through the consent and export mechanisms.

## 8. Log Sanitization Policy

### 8.1 Structured Logging

The system uses `shared.logging.logging_config.configure_logging()` for centralized logging configuration. All components use structured logging with named loggers (e.g., `logging.getLogger("vqa-perception")`).

### 8.2 PII Protection

- **No PII in log fields**: Structured logging separates machine-readable fields (timestamps, component names, metrics) from user content. Log fields should not contain user transcripts, scene descriptions, or personal information.
- **API key redaction**: API keys must not appear in log output. Environment variables are loaded via `os.environ.get()` and should never be logged at any level.
- **Telemetry events**: The `log_event()` function emits structured telemetry events with predefined fields. Custom event data should be limited to performance metrics and component status.

### 8.3 Log Storage

- Logs are stored locally in `.runtime/logs/`.
- No log forwarding to external services is implemented.
- Log files should be rotated to prevent unbounded disk growth.
- Log files are not encrypted at rest (consistent with session log storage).

### 8.4 Gaps

- No automated PII detection or scrubbing in log output.
- Bare `except` clauses in `core/memory/rag_reasoner.py` may inadvertently log user data in error messages.
- No log retention policy or automated rotation configuration documented.
