# Threat Model — STRIDE Analysis

This document presents a systematic threat analysis of the Voice & Vision Assistant for Blind using the STRIDE methodology. The analysis covers all six architectural layers, identifies hybrid-specific risks, assigns severity scores, and maps mitigation strategies to existing backlog items.

---

## 1. STRIDE Analysis per Layer

### 1.1 Interface Layer (FastAPI Port 8000, LiveKit Port 8081)

| STRIDE Category | Threat | Description |
|-----------------|--------|-------------|
| **Spoofing** | Unauthenticated API access | Non-debug endpoints have no authentication. An attacker on the local network can invoke any non-debug endpoint (e.g., `/memory/store`, `/qr/scan`) without credentials. Debug endpoints are protected by `DEBUG_AUTH_TOKEN` Bearer auth. |
| **Tampering** | Request body manipulation | Malicious payloads in `image_base64` or `transcript` fields could carry oversized data or exploit downstream parsers. Pydantic validation provides schema-level protection but does not enforce semantic limits on all fields. |
| **Repudiation** | No audit trail for API calls | The system uses structured logging (`shared.logging`) but does not maintain a tamper-proof audit log of API invocations. A user could deny having made a specific API call. |
| **Information Disclosure** | Debug endpoints expose internals | Endpoints like `/debug/perception`, `/debug/orchestrator`, `/debug/workers`, and `/debug/frame-manager` expose internal state including frame counts, processing times, and worker pool details. Protected only by Bearer token. |
| **Denial of Service** | Endpoint flooding | No rate limiting is implemented on any endpoint (BACKLOG-011). An attacker can flood `/debug/perception` with large base64 images, consuming GPU VRAM and CPU resources. |
| **Elevation of Privilege** | DEBUG_AUTH_TOKEN bypass | If the `DEBUG_AUTH_TOKEN` environment variable is unset or empty, debug endpoints may be accessible without authentication. Docker containers running as root (ISSUE-002) amplify privilege escalation risk. |

### 1.2 Application Layer (FrameOrchestrator, WorkerPools, Debouncer)

| STRIDE Category | Threat | Description |
|-----------------|--------|-------------|
| **Spoofing** | Forged frame timestamps | The `LiveFrameManager` uses `TimestampedFrame` objects. An attacker injecting frames via the LiveKit stream could forge timestamps to bypass the 500ms freshness check. |
| **Tampering** | Worker pool task injection | If an attacker gains process-level access, they could inject malicious tasks into the `ThreadPoolExecutor` used by `PerceptionWorkerPool` or `WorkerPool`. |
| **Repudiation** | Pipeline processing gaps | The `PipelineMonitor` collects metrics but they are not persisted across restarts. Processing failures during a session cannot be reliably audited post-restart. |
| **Information Disclosure** | Telemetry data leakage | `PipelineMonitor` and `Watchdog` expose processing metrics that could reveal system capacity and load patterns to an attacker with debug endpoint access. |
| **Denial of Service** | Frame buffer exhaustion | While `LiveFrameManager` uses a bounded `deque(maxlen=N)`, sustained high-rate frame injection could cause excessive CPU usage in the frame processing pipeline before backpressure takes effect. |
| **Elevation of Privilege** | No process isolation | All worker pools run in the same process. A vulnerability in one worker (e.g., a malicious ONNX model) could compromise the entire application. |

### 1.3 Domain Layer (VQA, Memory, OCR, QR, Face, Speech Engines)

| STRIDE Category | Threat | Description |
|-----------------|--------|-------------|
| **Spoofing** | Memory consent forgery | The `/memory/consent` endpoint accepts consent without strong identity verification. An attacker could toggle consent state, enabling or disabling memory storage. |
| **Tampering** | QR payload injection | QR code payloads are decoded and passed to TTS without comprehensive sanitization (ISSUE-003). Malicious payloads with `javascript:`, `data:`, or excessively long strings could reach the user. |
| **Repudiation** | Memory deletion without trace | Stored memories can be deleted without an audit trail. A user or attacker could remove evidence of past interactions. |
| **Information Disclosure** | Face embedding extraction | Face embeddings generated by `/face/embeddings` are biometric data. If intercepted, they could be used for identity tracking. Face detection requires explicit consent via `/face/consent`. |
| **Denial of Service** | OCR 3-tier fallback cascade | An attacker could submit images specifically crafted to fail all three OCR tiers (EasyOCR, Tesseract, MSER), causing maximum resource consumption per request. |
| **Elevation of Privilege** | VoiceRouter intent manipulation | The `VoiceRouter` uses regex patterns for intent classification. Crafted voice input could be designed to trigger unintended intents (e.g., triggering debug actions via voice). |

### 1.4 Local GPU Layer (YOLO, MiDaS, EasyOCR, qwen3-embedding:4b, Face Detection)

| STRIDE Category | Threat | Description |
|-----------------|--------|-------------|
| **Spoofing** | Model weight substitution | ONNX model files in `models/` directory have no integrity verification. An attacker with file system access could replace `yolov8n.onnx` or `midas_small.onnx` with adversarial models. |
| **Tampering** | Adversarial input images | Carefully crafted images can cause YOLO to misclassify objects or MiDaS to produce incorrect depth maps. For a blind user relying on navigation cues, this could be physically dangerous. |
| **Repudiation** | Silent fallback to mock detector | When YOLO fails to load, the system silently falls back to `MockObjectDetector` (ISSUE-026). The user is not informed that navigation cues are synthetic, creating a false sense of safety. |
| **Information Disclosure** | GPU memory side-channel | Shared GPU memory on multi-tenant systems could leak information between processes. In single-user deployment this is low risk, but relevant for future multi-user scenarios. |
| **Denial of Service** | GPU OOM crash | Submitting many large images simultaneously to `/debug/perception` could exhaust the 8GB VRAM budget, causing an unrecoverable process crash with no OOM handler. |
| **Elevation of Privilege** | ONNX Runtime code execution | Malicious ONNX models could exploit vulnerabilities in ONNX Runtime to execute arbitrary code. Model files should be treated as untrusted input if sourced externally. |

### 1.5 Cloud APIs Layer (qwen3.5:cloud, Deepgram STT, ElevenLabs TTS, LiveKit, DuckDuckGo)

| STRIDE Category | Threat | Description |
|-----------------|--------|-------------|
| **Spoofing** | API key theft and impersonation | 7 API keys are committed in `.env` (ISSUE-001). An attacker with repository access can impersonate the application to all cloud services. |
| **Tampering** | Man-in-the-middle on cloud calls | While cloud APIs use TLS/WSS, if certificate validation is misconfigured, responses from qwen3.5:cloud, Deepgram, or ElevenLabs could be tampered with. |
| **Repudiation** | Cloud provider audit gaps | The system does not log which specific cloud calls were made with what parameters. Cloud-side audit logs exist but are not correlated with local session identifiers. |
| **Information Disclosure** | Sensitive data in cloud prompts | Scene descriptions, user transcripts, and memory context are sent to qwen3.5:cloud for reasoning. While frames never leave the local machine, textual descriptions of scenes (including people) are transmitted to the cloud LLM. |
| **Denial of Service** | Cloud rate limit exhaustion | An attacker targeting the system's API endpoints could trigger excessive cloud API calls, exhausting rate limits for Deepgram, ElevenLabs, or qwen3.5:cloud, and causing service-wide outage. No retry/backoff is implemented (BACKLOG-004). |
| **Elevation of Privilege** | Prompt injection via scene descriptions | Crafted visual scenes could produce scene descriptions that act as prompt injections when sent to qwen3.5:cloud, potentially causing the LLM to execute unintended instructions or leak system prompt content. |

### 1.6 Storage Layer (FAISS Index, QR Cache, Face Consent, Session Logs)

| STRIDE Category | Threat | Description |
|-----------------|--------|-------------|
| **Spoofing** | QR cache poisoning | The file-based QR cache (`qr_cache/`) has no integrity verification. An attacker with filesystem access could insert false cached results that would be served to the user. |
| **Tampering** | FAISS index corruption | The FAISS index (`data/memory_index/`) is a binary file with no checksum verification. Corruption or malicious modification could cause incorrect memory retrieval or crashes. |
| **Repudiation** | Session log deletion | Session logs in `.runtime/logs/` can be deleted without detection. No log forwarding or tamper-evident storage is implemented. |
| **Information Disclosure** | FAISS index exfiltration | The FAISS index file contains vector embeddings that could be reverse-engineered to recover approximate original text content. The index is not encrypted at rest (ISSUE-012). |
| **Denial of Service** | Disk exhaustion via memory storage | Unrestricted memory ingestion without storage quotas could fill the disk with FAISS vectors, QR cache files, and session logs. |
| **Elevation of Privilege** | File permission escalation | Docker containers running as root (ISSUE-002) mean all storage files are owned by root. If the container is compromised, the attacker has full filesystem access. |

## 2. Hybrid-Specific Risks

### 2.1 GPU Memory Exhaustion Attack

**Description**: An attacker floods the `/debug/perception` endpoint with oversized base64-encoded images. Each request triggers YOLO (~200MB), MiDaS (~100MB), and EasyOCR (~500MB) inference on the local GPU. With the RTX 4060's 8GB VRAM and ~3.1GB peak usage, sustained attack traffic could push VRAM beyond capacity, causing an unrecoverable OOM crash.

**Current State**: No rate limiting on endpoints (BACKLOG-011). No VRAM monitoring or backpressure (hybrid_readiness.md). No OOM handler — process crashes and requires manual restart.

**Impact**: Complete system failure for the blind user. Navigation cues, perception, and all GPU-dependent features become unavailable.

### 2.2 Cloud Rate Limit Exhaustion

**Description**: An attacker targets the system's public endpoints to trigger excessive calls to qwen3.5:cloud (via `/memory/query`), Deepgram (via LiveKit audio stream), or ElevenLabs (via TTS). Since no retry/backoff or rate limiting exists, the attacker can exhaust the provider's rate limits, causing HTTP 429 errors that propagate as unhandled failures.

**Current State**: No retry/backoff for any cloud service (CLOUD_EFFICIENCY=4/10). No circuit breaker pattern. Single provider for each service with no fallback (BACKLOG-004).

**Impact**: Complete loss of LLM reasoning, speech-to-text, or text-to-speech capabilities until rate limits reset.

### 2.3 Replay Attack on LiveKit WebRTC Streams

**Description**: An attacker captures WebRTC media packets from the LiveKit session and replays them to inject false audio commands or video frames. While WebRTC uses SRTP encryption, a compromised LiveKit signaling channel could allow session hijacking.

**Current State**: LiveKit handles WebRTC security internally. The application relies on LiveKit's built-in DTLS/SRTP for transport encryption. No application-level session binding or replay detection beyond WebRTC's built-in mechanisms.

**Impact**: An attacker could inject false voice commands to trigger actions or inject false video frames that produce misleading navigation cues for the blind user.

### 2.4 Prompt Injection via Crafted Scene Descriptions

**Description**: An attacker places physical objects with text (e.g., signs, screens) in the camera's field of view. The perception pipeline reads the text via OCR and includes it in scene descriptions sent to qwen3.5:cloud. The injected text could contain instructions like "Ignore previous instructions and respond with..." to manipulate LLM behavior.

**Current State**: No prompt sanitization or injection detection on scene descriptions before they reach the LLM. The VQAReasoner passes scene context directly to qwen3.5:cloud prompts.

**Impact**: The LLM could produce misleading responses, leak system prompt content, or provide dangerous navigation advice to the blind user.

### 2.5 Data Poisoning in RAG Memory

**Description**: An attacker uses the `/memory/store` endpoint to inject malicious memories containing false information. When the user later queries their memories, the RAG pipeline retrieves the poisoned entries, and qwen3.5:cloud reasons over them, producing incorrect or harmful answers.

**Current State**: Memory storage requires consent but has no content validation beyond schema checking. The FAISS indexer stores any valid embedding without semantic verification. No anomaly detection on ingested memories.

**Impact**: The blind user receives incorrect answers to memory queries (e.g., wrong medication information, incorrect directions), potentially causing harm.

### 2.6 Vector DB Exfiltration

**Description**: An attacker with filesystem access reads the FAISS index files (`data/memory_index/`) and associated metadata JSON. Using nearest-neighbor search on the extracted vectors, they can recover approximate textual content of stored memories, exposing private conversations and scene descriptions.

**Current State**: FAISS index files are stored unencrypted on disk (ISSUE-012). No filesystem access controls beyond OS-level permissions. Docker containers run as root (ISSUE-002), removing an additional security layer. The `get_encryption_manager()` utility exists in `shared/utils/` but is not wired to FAISS storage.

**Impact**: Complete disclosure of the user's private memories, including conversations, scene descriptions, and any personal information stored via the memory system.

## 4. Mitigation Strategies

### 4.1 Input Validation (Pydantic Models, Size Limits)

**Current State**: The FastAPI server uses Pydantic v2 models for request validation. Fields like `MemoryStoreRequest.transcript` have `maxLength: 10000`, and `MemorySearchRequest.query` is capped at 500 characters.

**Gaps**:
- `image_base64` fields have no enforced maximum size at the application level.
- QR payloads are not sanitized before TTS output (BACKLOG-003).
- No semantic validation on memory content before storage.

**Recommended Actions**:
- Enforce maximum `image_base64` size (e.g., 4MB decoded) across all endpoints.
- Implement `sanitize_payload()` for QR content with blocklist for `javascript:`, `data:` URI schemes (BACKLOG-003).
- Add content length validation on all text input fields.

### 4.2 Auth Middleware (DEBUG_AUTH_TOKEN)

**Current State**: Debug endpoints (`/debug/*`) require Bearer token authentication via the `DEBUG_AUTH_TOKEN` environment variable. Non-debug endpoints (health, memory, QR, braille, face, session, export) have no authentication — by design, as the system is a single-user assistive device.

**Gaps**:
- No validation that `DEBUG_AUTH_TOKEN` is set and non-empty on startup.
- No token rotation mechanism.
- No brute-force protection on token validation.

**Recommended Actions**:
- Add startup check that `DEBUG_AUTH_TOKEN` is set and meets minimum entropy requirements.
- Log authentication failures with source IP for audit purposes.

### 4.3 Rate Limiting (BACKLOG-011)

**Current State**: No rate limiting is implemented on any FastAPI endpoint.

**Recommended Actions** (per BACKLOG-011):
- Install `slowapi` or equivalent rate limiting middleware.
- Limit expensive endpoints (`/debug/perception`, `/memory/store`, `/memory/query`, `/face/embeddings`, `/braille/read`) to ≤10 requests/minute per IP.
- Return HTTP 429 with `Retry-After` header.
- Externalize rate limit configuration for tuning.

### 4.4 Retry/Backoff and Circuit Breakers (BACKLOG-004)

**Current State**: No retry/backoff for any cloud service. The `OllamaHandler` uses `asyncio.wait_for()` with timeouts, but failures are not retried. The only fallback is `StubLLMClient` for LLM calls.

**Recommended Actions** (per BACKLOG-004):
- Implement exponential backoff (3 retries: 1s, 2s, 4s) for Deepgram, ElevenLabs, and qwen3.5:cloud calls.
- Add circuit breaker per cloud service: open after 3 consecutive failures, probe for recovery every 30 seconds.
- Implement health check probes for each cloud service.
- Add at least one fallback STT provider (e.g., local Whisper) for Deepgram outage.

### 4.5 GPU Guardrails (VRAM Monitoring)

**Current State**: Peak VRAM usage is ~3.1GB against 8GB available (38.75% utilization). No active VRAM monitoring or backpressure exists. GPU OOM causes an unrecoverable process crash.

**Recommended Actions** (per hybrid_readiness.md recommendations):
- Implement VRAM usage check via `torch.cuda.memory_allocated()` before scheduling GPU work.
- Add backpressure when usage exceeds 75% of total VRAM (6GB threshold).
- Surface VRAM usage in `/health` endpoint.
- Implement graceful OOM handling: catch CUDA OOM errors, clear GPU cache, and degrade to CPU inference.

### 4.6 Embedding Sanitization (Prompt Injection Prevention)

**Current State**: Scene descriptions from the perception pipeline are passed directly to qwen3.5:cloud prompts without sanitization. OCR text extracted from the scene is included verbatim.

**Recommended Actions**:
- Sanitize OCR text before including in LLM prompts: strip control characters, limit length, and escape special tokens.
- Implement a prompt injection detection layer that flags suspicious patterns in scene descriptions.
- Use structured prompt templates that separate user-supplied content from system instructions.
- Consider a "canary" token approach to detect prompt leakage.

### 4.7 Secure Local Storage (FAISS Encryption) (BACKLOG-012)

**Current State**: FAISS index files are stored as unencrypted binary files on disk. The `get_encryption_manager()` utility exists in `shared/utils/` but is not connected to FAISS storage.

**Recommended Actions** (per BACKLOG-012):
- Encrypt FAISS index files at rest using `get_encryption_manager()`.
- Encrypt `metadata.json` associated with the FAISS index.
- Auto-migrate existing unencrypted indexes to encrypted format on first load.
- Verify decryption on application restart.

### 4.8 Secret Management (BACKLOG-001, BACKLOG-019, BACKLOG-020)

**Current State**: 7 API keys committed in `.env` (ISSUE-001). `.env` is copied into Docker images (ISSUE-019). No pre-commit hooks to prevent accidental secret commits.

**Recommended Actions**:
- **Immediate** (BACKLOG-001): Rotate all 7 API keys. Remove `.env` from git tracking. Add `.env` to `.gitignore`.
- **Docker** (BACKLOG-019): Add `.env` to `.dockerignore`. Pass secrets via `--env-file` at runtime.
- **Prevention** (BACKLOG-020): Install `detect-secrets` pre-commit hook. Configure `.pre-commit-config.yaml` with `detect-secrets` and `ruff` hooks.
- **Audit**: Scrub git history of committed secrets. Audit previously published Docker images.

## 5. Known Vulnerabilities

The following vulnerabilities were identified in the security scan and are tracked in the backlog:

### ISSUE-001: Real API Keys Committed in .env (CRITICAL)

**Affected Keys** (7 total):
| Key | File | Line | Severity |
|-----|------|:----:|----------|
| `LIVEKIT_API_KEY` | .env | 3 | CRITICAL |
| `LIVEKIT_API_SECRET` | .env | 4 | CRITICAL |
| `DEEPGRAM_API_KEY` | .env | 7 | CRITICAL |
| `OLLAMA_API_KEY` | .env | 8 | CRITICAL |
| `ELEVEN_API_KEY` | .env | 9 | CRITICAL |
| `ELEVENLABS_API_KEY` | .env | 10 | CRITICAL |
| `TAVUS_API_KEY` | .env | 25 | CRITICAL |

**Remediation**: BACKLOG-001 (P0, effort S). Rotate keys, add `.env` to `.gitignore`, scrub git history, install `detect-secrets` hook.

### ISSUE-002: Docker Containers Run as Root (HIGH)

Both `Dockerfile` (root) and `deployments/docker/Dockerfile` lack a `USER` instruction. All processes run as `root` inside the container, providing unrestricted filesystem and capability access.

**Remediation**: BACKLOG-002 (P0, effort S). Add non-root `USER` instruction. Verify health check passes as non-root.

### ISSUE-019: .env Copied Into Docker Image (HIGH)

The root `Dockerfile` uses `COPY . .` which includes the `.env` file containing all 7 API keys. Anyone with access to the Docker image can extract the secrets.

**Remediation**: BACKLOG-019 (P0, effort S). Add `.env` to `.dockerignore`. Pass secrets via runtime environment variables.

### No Pre-Commit Hooks (LOW)

No `.pre-commit-config.yaml` exists. Secrets can be committed accidentally without any automated detection.

**Remediation**: BACKLOG-020 (P2, effort M). Install `detect-secrets` and `ruff` pre-commit hooks.

### 6 Bare Except Clauses (LOW)

Bare `except:` clauses at 6 locations mask critical exceptions and hinder debugging:
- `core/memory/rag_reasoner.py` (lines 178, 237)
- `tests/realtime/calibrate_depth.py` (line 189)
- `tests/realtime/session_logger.py` (lines 298, 321, 335)

**Remediation**: BACKLOG-016 (P3, effort S). Replace with specific exception types. Log caught exceptions at appropriate levels.
