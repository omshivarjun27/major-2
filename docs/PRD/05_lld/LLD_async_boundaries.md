# Async and Sync Boundaries

### 1. Async vs Sync Boundaries

| Component | Async? | Correct? | Notes |
|-----------|--------|----------|-------|
| OllamaHandler (LLM) | Yes | ✅ | Uses httpx.AsyncClient with asyncio.wait_for() |
| OllamaEmbedder (embedding) | No | ❌ | Synchronous HTTP calls block the event loop (ISSUE-022) |
| OllamaClient (memory LLM) | Yes | ✅ | Async HTTP via Ollama cloud runtime |
| Deepgram STT | Yes | ✅ | WebSocket streaming managed by LiveKit plugin |
| ElevenLabs TTS | Yes | ✅ | HTTP/WebSocket managed by LiveKit plugin |
| YOLODetector.detect() | No | ⚠️ | CPU-bound; must use run_in_executor() |
| MiDaSDepthEstimator.estimate() | No | ⚠️ | CPU-bound; must use run_in_executor() |
| PerceptionWorkerPool | ThreadPool | ✅ | ThreadPoolExecutor for CPU-bound inference |
| FAISSIndexer | No | ⚠️ | Uses RLock and blocks during disk I/O |

### 2. Cloud API Call Boundaries
- **OllamaHandler → qwen3.5:cloud**: Performs async HTTP POST requests. Timeouts are enforced via `asyncio.wait_for()`. No automatic retry logic is currently implemented.
- **Deepgram → STT**: Utilizes a persistent WebSocket stream. Reconnection logic is handled internally by the LiveKit plugin.
- **ElevenLabs → TTS**: Uses a mix of HTTP and WebSocket streaming for audio generation.
- **DuckDuckGo → Search**: Executes async HTTP requests. No structured error handling or fallback is documented.
- **Tavus → Avatar**: Interfaces via async HTTP for configuration and WebSockets for real-time video stream management.

### 3. WorkerPool Execution Boundaries
- **PerceptionWorkerPool**: Uses a dedicated ThreadPoolExecutor. It parallelizes object detection, depth estimation, OCR, and embedding generation. The pool size is bounded by worker count to prevent CPU starvation.
- **WorkerPool**: A generic ThreadPoolExecutor used for offloading various short-lived, blocking tasks from the main event loop.
- **Resource Constraints**: Pools are bounded by thread count, but the system lacks explicit VRAM-aware scheduling for GPU-intensive tasks.

### 4. Frame Processing Lifecycle
The following sequence describes the lifecycle of a video frame within the system:
1. WebRTC VideoFrame arrives via the LiveKit Agent.
2. The Agent pushes the frame to the LiveFrameManager (a bounded deque that drops the oldest frames if full).
3. LiveFrameManager wraps the raw frame in a `TimestampedFrame` object with a unique ID and timestamp.
4. FrameOrchestrator performs a freshness check (max_age=500ms) and rejects stale frames.
5. FrameOrchestrator dispatches parallel tasks to the PerceptionWorkerPool:
   - `detect(image)` → List of Detections.
   - `estimate_depth(image)` → DepthMap.
   - `ocr_read(image)` → OCR text (if enabled).
6. Results are assembled into a single `PerceptionResult` object.
7. SceneGraphBuilder builds a SceneGraph from the perception data, calculating spatial relations.
8. SpatialFuser fuses the latest perception result with historical tracks (EMA smoothing).
9. MicroNavFormatter formats the fused results into a concise TTS cue string.
10. Debouncer checks if the cue should be suppressed based on a 7-second window.
11. If not debounced, the StreamingTTSCoordinator dispatches text to ElevenLabs; audio chunks are streamed back to the user via LiveKit.

### 5. Streaming Pipeline Lifecycle
The voice interaction pipeline follows this sequence:
1. Raw audio frames arrive via the LiveKit WebRTC track.
2. The Deepgram STT plugin processes the stream via WebSocket and returns real-time transcripts.
3. Transcripts are delivered to the AllyVisionAgent.
4. VoiceRouter routes the transcript to a specific intent (e.g., VISUAL_DESCRIBE) and handler.
5. The specific handler is dispatched (e.g., capture a frame and run VQA or perform an internet search).
6. Response text is generated by the LLM (qwen3.5:cloud) or formatted from perception results.
7. StreamingTTSCoordinator buffers the text at sentence boundaries.
8. ElevenLabs synthesizes audio chunks via HTTP/WebSocket.
9. Audio chunks are delivered back to the user via the LiveKit WebRTC audio track.

### 6. Backpressure Control
The system implements several mechanisms to manage load and prevent resource exhaustion:
- **LiveFrameManager**: Uses a `collections.deque` with a fixed `maxlen`. New frames overwrite the oldest, ensuring the buffer never grows unboundedly.
- **Debouncer**: Uses a 7-second time-based window to suppress repetitive identical navigation cues.
- **Worker Pools**: Both PerceptionWorkerPool and WorkerPool are bounded by a fixed number of threads.
- **TemporalFilter**: Caps the number of tracked objects at 20, evicting the lowest-confidence tracks first.
- **Frame Freshness**: Frames older than 500ms are discarded by the orchestrator.
- **Gap**: There is currently no backpressure control for cloud API requests; they are dispatched as they arrive without queue depth monitoring.

### 7. Queue Overflow Handling
- **LiveFrameManager Deque**: Silently drops old frames. This prioritizes low latency over data completeness.
- **ThreadPoolExecutor Queues**: The task queues for the worker pools can grow if the processing rate is slower than the arrival rate. This presents a potential memory risk under sustained heavy load.
- **FAISS Index**: Enforces a `max_vectors` limit (default 5000) with an eviction policy based on time or LRU.
- **QR Cache**: Uses file-based storage with TTL-based expiry (86400s) to prevent uncontrolled growth of the cache directory.
- **Gap**: The memory ingestion pipeline lacks explicit queue monitoring, which could lead to memory issues if summaries and embeddings are generated slower than they are requested.

### 8. Circuit Breaker Recommendation
The following circuit breaker strategy is recommended to improve cloud resilience:
- **Current State**: No circuit breakers are implemented; system relies on static stubs or manual recovery.
- **Pattern**: Implement a per-service state machine (CLOSED → OPEN → HALF_OPEN).
- **Service-Specific Recommendations**:
  - **Deepgram (STT)**: Open after 3 consecutive WebSocket failures; probe every 30s.
  - **ElevenLabs (TTS)**: Open after 3 consecutive HTTP failures; probe every 30s.
  - **Ollama Cloud (LLM)**: Open after 3 consecutive timeouts; fallback to `StubLLMClient` and probe every 30s.
  - **DuckDuckGo (Search)**: Open after 5 failures; probe every 60s.
  - **Tavus (Avatar)**: Since this is an optional service, open after 3 failures and disable gracefully.
- **Implementation**: It is recommended to use a lightweight decorator on the infrastructure adapters to manage these states.
